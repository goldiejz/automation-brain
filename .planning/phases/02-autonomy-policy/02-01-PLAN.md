Refine the existing `scripts/ark-policy.sh` into the AOS foundation: add a cascading config loader, factor reusable bits into `scripts/lib/policy-config.sh`, harden the audit log schema (so Phase 3's observer-learner can consume it without breaking changes), and keep the existing decision functions / self-test green.

Output: a sourceable policy library + cascading config resolver + JSONL audit log with a stable, Phase-3-ready schema and high-entropy decision_id.

**W-6 fix:** The previously-locked schema (`{ts, schema_version, class, decision, reason, context}`) had no way for Phase 3 to correlate or patch decisions. This plan adds three forward-looking fields NOW (still schema_version=1) so Phase 2 ships data Phase 3 can directly consume:
- `decision_id` — unique ID per log line (ts-based + 64-bit random suffix). Lets Phase 3 patch outcome retroactively.
- `outcome` — `null` on write (Phase 2 doesn't know yet); Phase 3 patches via decision_id.
- `correlation_id` — optional, links related decisions (e.g., a budget reset caused by a dispatch failure). Caller passes through; null if not provided.

**NEW-W-4 fix:** The previous draft's decision_id format `${ts_compact}-${RANDOM*RANDOM%1000000}` collapsed to ~20 effective bits → 6 decimal digits. Under stress (Tier 8 fires 5–10 audit writes/sec), birthday-paradox collision probability is small but non-zero. Phase 3's patch-by-decision_id contract REQUIRES real uniqueness. Fix: replace random suffix with 16 hex chars from `/dev/urandom` (64-bit entropy, macOS-safe via `od -An -tx8`). Add a Tier 8 stress check (delegated to 02-08) that fires 100 _policy_log calls in a tight loop and asserts 100 distinct decision_id values.
</objective>

Audit log line schema (LOCKED — Phase 3 reads this — schema_version=1):

```json
{
  "ts": "<ISO8601 UTC>",
  "schema_version": 1,
  "decision_id": "<YYYYMMDDTHHMMSSZ>-<16-hex-chars>",
  "class": "<budget|dispatch|zero_tasks|dispatch_failure|escalation|self_heal>",
  "decision": "<UPPER_SNAKE>",
  "reason": "<machine-readable_token_pattern>",
  "context": <json-object-or-null>,
  "outcome": null,
  "correlation_id": null
}
```

Field semantics:
- `ts` — ISO8601 UTC timestamp (e.g., `2026-04-26T14:23:45Z`)
- `schema_version` — integer `1`
- `decision_id` — globally unique. Format: compact-ts (`%Y%m%dT%H%M%SZ`) + `-` + **16 hex chars from `/dev/urandom`** (64-bit entropy). Generation (macOS-safe): `head -c 8 /dev/urandom | od -An -tx8 | tr -d ' \n'`. Phase 3 uses this to patch outcome retroactively.
- `class` — one of the listed enum values
- `decision` — UPPER_SNAKE_CASE return value of the decision function
- `reason` — machine-readable token pattern (no free text; greppable)
- `context` — JSON object with decision-specific data (input args, derived values), or `null`
- `outcome` — ALWAYS written as `null` in Phase 2. Phase 3's observer-learner patches this via `jq` rewrites keyed on `decision_id`. Possible values Phase 3 will set: `"success" | "failure" | "ambiguous"`.
- `correlation_id` — `null` by default. When a decision is causally linked to a prior one (e.g., `policy_budget_decision` returns AUTO_RESET → caller invokes `ark-budget.sh --reset` which then logs a follow-up event), the caller passes the prior `decision_id` so Phase 3 can reconstruct chains.

The shared-lib pattern to mirror is `scripts/lib/gsd-shape.sh` (sourceable, self-test on `$1=test`, no top-level side effects beyond function defs and a couple of safe `mkdir -p`).
